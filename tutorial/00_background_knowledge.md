## 1. 什么是性能模拟？为什么 LLM 需要它？

### 1.1 性能模拟的定义

**性能模拟**（Performance Simulation）是指在**不实际运行**完整计算任务的前提下，通过分析模型和硬件特性来**估计**程序的执行时间。

打一个比方：你想知道从北京开车到上海需要多长时间，但你并不需要真的开一趟。你可以根据距离、限速、路况等信息来**估算**总时间。性能模拟做的就是类似的事情——根据"计算量"、"内存带宽"、"网络延迟"等信息来估算神经网络训练或推理的耗时。

### 1.2 为什么 LLM 特别需要性能模拟？

大语言模型（LLM）的规模极其庞大。以 GPT-3 为例，它拥有 1750 亿个参数，单次训练可能需要数千块 GPU 运行数周之久。在这种量级下，以下问题变得至关重要：

- **硬件选型**：在购买或租用 GPU 之前，需要知道不同硬件上的预期训练时间。
- **模型架构探索**：在构建新模型时，希望快速评估不同架构选择对训练效率的影响。
- **资源调度**：在集群管理中，需要预估任务耗时来优化资源分配。
- **分布式策略选择**：不同的并行策略（数据并行、张量并行等）会带来截然不同的性能表现，需要在实际部署前做出决策。

如果每次评估都需要真正运行模型，代价将非常高昂。性能模拟器允许我们在**几秒到几分钟内**得到估算结果，而非等待数小时甚至数天。

### 1.3 SysSim 的设计哲学

SysSim 采用**混合方法**：将基于物理的分析模型（Roofline）与基于机器学习的效率预测相结合。

- **Roofline 模型**提供了基于硬件规格的理论上限——这部分是确定性的、可解释的。
- **ML 效率预测器**学习实际执行中与理论值之间的偏差——捕获缓存效应、GPU 利用率、内核启动开销等难以用公式表达的因素。

两者结合的核心公式非常简洁：

$$T_{\text{actual}} = \frac{T_{\text{roofline}}}{\eta}$$

其中 \(T_{\text{roofline}}\) 是分析模型给出的理论最短时间，\(\eta \in (0, 1]\) 是 ML 模型预测的效率因子。

## 2. 计算图与有向无环图（DAG）

### 2.1 什么是计算图？

在深度学习中，**计算图**（Computational Graph）是用来描述一系列数学运算之间的数据流关系的图结构。每个节点代表一个运算（如矩阵乘法、激活函数），每条边代表数据（张量）在运算之间的传递关系。

举一个最简单的例子——线性层加 ReLU 激活：

```
输入 x
  │
  ▼
┌──────────┐
│ 矩阵乘法  │    y = W * x + b
└────┬─────┘
     │
     ▼
┌──────────┐
│   ReLU   │    z = max(0, y)
└────┬─────┘
     │
     ▼
  输出 z
```

在这个图中，ReLU 依赖于矩阵乘法的输出。这种"先后关系"就是计算图所表达的核心信息。

### 2.2 有向无环图（DAG）

**有向无环图**（Directed Acyclic Graph, DAG）是计算图的数学形式化表述：

- **有向**：边有方向，表示数据流动的方向（从生产者到消费者）。
- **无环**：图中不存在从某个节点出发经过若干条边后又回到该节点的路径。

DAG 的"无环"性质非常关键——它保证了计算可以按照拓扑排序的顺序执行，不会出现循环依赖。

```
     A ──────► B ──────► D
     │                    ▲
     │                    │
     └──────► C ──────────┘

  拓扑排序：A → B → C → D  或  A → C → B → D
  （两种都合法，因为 B 和 C 之间没有依赖关系）
```

### 2.3 关键路径

在 DAG 中，**关键路径**（Critical Path）是指从起始节点到终止节点的最长路径。它决定了整个计算图的最短完成时间——即使我们有无限的并行资源，也不可能比关键路径更快地完成所有计算。

```
     A (2ms) ──► B (5ms) ──► D (1ms)
     │                        ▲
     │                        │
     └──────► C (3ms) ────────┘

  路径 1: A → B → D = 2 + 5 + 1 = 8ms  ← 关键路径
  路径 2: A → C → D = 2 + 3 + 1 = 6ms

  最短完成时间 = 8ms（由关键路径决定）
```

### 2.4 SysSim 中的计算图

SysSim 在追踪（Trace）PyTorch 模型执行时，会构建一个算子级别的 DAG。图中的每个节点是一个算子（Operator），拥有以下属性：

- **算子类型**：GEMM（矩阵乘法）、ATTN（注意力）、MATH（通用数学运算）、COLLECTIVE（集合通信）、MEMORY（内存拷贝）、BARRIER（全局同步）、STREAM_SYNC（流间同步）
- **预估执行时间**：通过 Roofline + ML 效率模型计算
- **依赖关系**：数据依赖（谁的输出是谁的输入）和流依赖（跨 CUDA 流的同步）

通过对这个 DAG 进行关键路径分析，SysSim 可以估计整个模型训练或推理的总耗时。

## 3. Roofline 模型：计算瓶颈 vs 内存瓶颈

### 3.1 直觉理解

想象你在工厂的流水线上工作。流水线有两个环节：

1. **搬运工**负责把原材料搬到机器旁边（类比：内存读写）
2. **机器**负责加工原材料（类比：计算单元做运算）

流水线的整体速度取决于哪个环节更慢：

- 如果机器加工很快，但搬运工搬运很慢，那瓶颈在搬运——这就是**内存瓶颈**（memory-bound）。
- 如果搬运工很快，但机器加工很慢，那瓶颈在加工——这就是**计算瓶颈**（compute-bound）。

### 3.2 Roofline 模型的数学表述

Roofline 模型为每个算子计算两个时间约束：

**计算约束**（Compute Constraint）：

$$T_{\text{compute}} = \frac{\text{FLOPs}}{\text{Peak FLOP/s}}$$

这表示：即使数据瞬间就位，纯粹的计算也需要这么长时间。

**内存约束**（Memory Constraint）：

$$T_{\text{memory}} = \frac{\text{Bytes}}{\text{Peak Bandwidth}}$$

这表示：即使计算瞬间完成，光是读写数据也需要这么长时间。

**Roofline 预估时间**：

$$T_{\text{roofline}} = \max(T_{\text{compute}},\ T_{\text{memory}})$$

取两者的最大值，因为一个算子的执行时间至少受限于更慢的那个因素。

### 3.3 算术强度

**算术强度**（Arithmetic Intensity）是判断一个算子是计算瓶颈还是内存瓶颈的关键指标：

$$\text{Arithmetic Intensity} = \frac{\text{FLOPs}}{\text{Bytes}}$$

单位是 FLOP/Byte，表示每传输一个字节需要做多少次浮点运算。

- 算术强度**高**的运算（如大矩阵乘法）通常是**计算瓶颈**：数据量相对少，但计算量大。
- 算术强度**低**的运算（如逐元素加法、LayerNorm）通常是**内存瓶颈**：计算简单，但需要读写大量数据。

### 3.4 Roofline 图示

```
性能
(FLOP/s)
  │
  │          ┌──────────────────── Peak Compute（计算天花板）
  │         ╱
  │        ╱
  │       ╱
  │      ╱    ← 这条斜线的斜率 = Peak Bandwidth
  │     ╱
  │    ╱
  │   ╱
  │  ╱
  │ ╱
  │╱
  └──────────────────────────────────────►
                 算术强度 (FLOP/Byte)
       ◄── 内存瓶颈区域 ──►◄── 计算瓶颈区域 ──►
```

在"屋顶"的拐角点左侧，性能受限于内存带宽（斜线部分）；在拐角点右侧，性能受限于计算峰值（水平部分）。

### 3.5 大小感知的峰值选择

GPU 的实际峰值性能因算子规模而异：

- **大规模运算**（所有维度 \(\geq 512\)）：使用峰值张量单元吞吐量。GPU 利用率高，内核启动开销可被摊销。
- **小规模运算**（任意维度 \(< 512\)）：使用保守峰值。GPU 并行度不足，内核启动开销（约 7 微秒）占比显著。

这是 SysSim 对标准 Roofline 模型的一个实用改进。

## 4. FLOP 计数

### 4.1 什么是 FLOP？

**FLOP**（Floating Point Operation）即"浮点运算次数"。它是衡量一个计算任务"计算量大小"的基本单位。

- 一次加法 = 1 FLOP
- 一次乘法 = 1 FLOP
- 一次乘加运算（Multiply-Accumulate, MAC）= 2 FLOPs（一次乘法 + 一次加法）

注意区分：
- **FLOP**（单数）：浮点运算次数（数量）
- **FLOP/s**（FLOPs per second）：每秒浮点运算次数（速率），用于衡量硬件性能

### 4.2 常见算子的 FLOP 公式

**矩阵乘法** \(C = A \times B\)，其中 \(A\) 的形状为 \(M \times K\)，\(B\) 的形状为 \(K \times N\)：

$$\text{FLOPs} = 2 \times M \times N \times K$$

因子 2 来源于每个输出元素需要 \(K\) 次乘法和 \(K-1\) 次加法（近似为 \(2K\) 次运算），共有 \(M \times N\) 个输出元素。

**注意力机制**（Scaled Dot-Product Attention），给定 batch size \(B\)、注意力头数 \(H\)、序列长度 \(S\)、头维度 \(D\)：

$$\text{FLOPs} \approx 2 \times B \times H \times S^2 \times D$$

注意力的计算量与序列长度的**平方**成正比，这也是为什么长序列处理在 LLM 中尤其昂贵的原因。

**逐元素运算**（如 ReLU、加法）：

$$\text{FLOPs} = \text{元素总数}$$

每个元素做一次运算。

**归一化运算**（如 LayerNorm）：

$$\text{FLOPs} \approx 5 \times \text{元素总数}$$

包含求均值、求方差、归一化、缩放和偏移等步骤。

### 4.3 FLOP 计数的意义

FLOP 计数是 Roofline 模型中计算约束 \(T_{\text{compute}}\) 的核心输入。准确的 FLOP 计数直接影响性能估计的准确性。

SysSim 为每种算子类型维护了一个 FLOP 计数注册表（Registry），确保对 PyTorch 中各类运算的 FLOP 计数是准确且一致的。

## 5. 机器学习效率预测

### 5.1 为什么仅靠 Roofline 不够？

Roofline 模型给出的是**理论性能上限**——假设 GPU 的每个计算单元和内存通道都在满负荷工作。但现实中，GPU 的实际性能往往远低于理论峰值，原因包括：

- **GPU 核心利用率不足**：小规模运算无法充分利用所有计算核心。
- **缓存命中率**：数据访问模式影响 L1/L2 缓存的命中率。
- **内核启动开销**：每次启动一个 GPU 内核都有固定的开销（约数微秒）。
- **线程束（Warp）调度效率**：GPU 的线程调度机制并非总是最优的。

这些因素使得纯 Roofline 预测往往**过于乐观**。

### 5.2 效率因子

SysSim 定义**效率因子** \(\eta\) 来表示实际性能与理论峰值之间的比率：

$$\eta = \frac{T_{\text{roofline}}}{T_{\text{measured}}} \in (0, 1]$$

- \(\eta = 1.0\)：完美效率，实际时间等于 Roofline 时间。
- \(\eta = 0.5\)：50% 效率，实际时间是 Roofline 时间的 2 倍。
- \(\eta = 0.1\)：10% 效率，实际时间是 Roofline 时间的 10 倍（常见于非常小的运算）。

效率通常与运算规模正相关：

```
效率 η
1.0 ┤
    │                              ●●●●●●●●
    │                         ●●●●
0.5 ┤                    ●●●●
    │              ●●●●●
    │         ●●●●
    │    ●●●●
0.1 ┤●●●
    │
0.0 ┤
    └──────────────────────────────────────►
          小运算                     大运算
              运算规模（矩阵维度等）
```

### 5.3 特征工程

ML 效率预测器不直接使用原始参数，而是精心设计的特征组合：

**基础特征**（对数缩放的维度信息）：
- 矩阵乘法：M、N、K 维度、纵横比、极值
- 注意力：batch、heads、序列长度、头维度
- 数学运算：batch、隐藏层大小、总元素数

**Roofline 包络特征**（瓶颈特性）：
- 各约束时间（对数缩放）
- 约束比率（距离瓶颈的远近）
- 主导约束（独热编码：是计算瓶颈还是内存瓶颈）

**硬件描述特征**：
- 容量比率（架构特征）
- 绝对容量（规模信息）

将 Roofline 的分析结果作为 ML 模型的输入特征，是 SysSim 混合方法的关键设计——它让 ML 模型了解到当前运算处于什么样的硬件约束环境中。

### 5.4 ML 模型架构

SysSim 支持两种 ML 后端：

**MLP（多层感知机）**：
- 4 个隐藏层，带批归一化
- Dropout 正则化
- Sigmoid 输出层（将预测约束在 \((0, 1]\) 范围内）

**XGBoost（梯度提升树）**：
- 基于直方图的树构建
- L1/L2 正则化、子采样
- 早停机制

两种模型都是**硬件特定**的——在 A100 上训练的效率模型不能直接用于 H100，因为不同 GPU 架构的缓存大小、SM 数量、调度特性等均不同。

### 5.5 回退机制

当 ML 模型不可用时，SysSim 优雅地降级为纯 Roofline 估计（\(\eta = 1.0\)）。这提供了保守但可解释的预测结果。

## 6. 集合通信

### 6.1 什么是集合通信？

在分布式训练中，多块 GPU 需要互相交换数据。**集合通信**（Collective Communication）是指一组预定义的通信模式，它们描述了多个参与者之间如何协调地发送和接收数据。

可以类比课堂场景：
- **Broadcast（广播）**：老师向全班同学宣布一件事——一对多通信
- **Reduce（归约）**：全班同学报出自己的成绩，老师算出平均分——多对一汇总
- **AllReduce（全归约）**：全班同学的成绩汇总后，每个同学都得到平均分——多对多汇总
- **AllGather（全收集）**：每个同学写下一段话，最后每个同学都拿到所有人的完整文集

### 6.2 八种集合通信原语

SysSim 支持以下八种集合通信操作：

```
┌────────────────────────────────────────────────────────────────┐
│  操作名称        描述                    数据流                │
├────────────────────────────────────────────────────────────────┤
│  AllReduce      全部归约后分发给所有人    全部 → 归约 → 全部   │
│  Broadcast      从一个根节点分发给所有人  一个 → 全部          │
│  Reduce         全部归约后汇总到一个人    全部 → 一个          │
│  AllGather      收集所有人的数据给所有人  全部 → 全部（拼接）  │
│  ReduceScatter  归约后将结果分散给各人    全部 → 归约 → 分散   │
│  AllToAll       每人给每人不同的数据      全部 ↔ 全部          │
│  Scatter        从一个人分发给所有人      一个 → 分散          │
│  Gather         从所有人收集到一个人      全部 → 一个（拼接）  │
└────────────────────────────────────────────────────────────────┘
```

### 6.3 AllReduce 的实现：环形算法

AllReduce 是分布式训练中最常用的通信操作（梯度同步），SysSim 使用**环形算法**（Ring Algorithm）实现它，分两步完成：

**步骤 1：ReduceScatter（归约-分散）**

```
初始状态（4 块 GPU，每块持有完整梯度向量的一部分用于归约）：

GPU 0: [A0] [A1] [A2] [A3]
GPU 1: [B0] [B1] [B2] [B3]
GPU 2: [C0] [C1] [C2] [C3]
GPU 3: [D0] [D1] [D2] [D3]

经过 3 轮环形传递和累加后：

GPU 0: [ABCD_0] [    ] [      ] [      ]
GPU 1: [      ] [ABCD_1] [    ] [      ]
GPU 2: [      ] [      ] [ABCD_2] [    ]
GPU 3: [      ] [      ] [      ] [ABCD_3]

每块 GPU 持有归约结果的 1/4。
```

**步骤 2：AllGather（全收集）**

```
再经过 3 轮环形传递后，每块 GPU 都拥有完整的归约结果：

GPU 0: [ABCD_0] [ABCD_1] [ABCD_2] [ABCD_3]
GPU 1: [ABCD_0] [ABCD_1] [ABCD_2] [ABCD_3]
GPU 2: [ABCD_0] [ABCD_1] [ABCD_2] [ABCD_3]
GPU 3: [ABCD_0] [ABCD_1] [ABCD_2] [ABCD_3]
```

环形算法的优势在于：每块 GPU 在任一时刻只与左右邻居通信，**带宽利用率最优**。

### 6.4 通信 DAG 与依赖关系

SysSim 将每个集合通信操作分解为一系列**点对点消息**，并自动推断依赖关系：

- **规则 1**：同一发送者的消息是顺序的（一个发送者不能同时发两条消息）
- **规则 2**：同一接收者的消息是顺序的（一个接收者不能同时接收两条消息）
- 无依赖的操作可以**并行**执行

## 7. LogGP 网络模型

### 7.1 为什么需要网络性能模型？

在分布式训练中，GPU 之间的通信时间可能占总训练时间的很大比例。为了准确模拟整体性能，我们需要一个能够估算点对点消息传输时间的模型。

### 7.2 LogGP 模型参数

LogGP 是一种广泛使用的**并行通信性能模型**，使用四个参数描述网络特性：

| 参数 | 名称 | 含义 |
|------|------|------|
| \(L\) | Latency（延迟） | 消息从发送端到接收端的单程传播延迟（秒） |
| \(o\) | Overhead（开销） | 每条消息在 CPU 端的处理时间（秒） |
| \(g\) | Gap per message（消息间隔） | 连续两条消息之间的最小时间间隔（秒） |
| \(G\) | Gap per byte（字节间隔） | 每字节的传输时间（秒/字节），即带宽的倒数 |

### 7.3 消息传输时间公式

一条大小为 \(m\) 字节的消息的传输时间为：

$$T(m) = L + 2o + (m - 1) \times G$$

其中：
- \(L\) 是网络延迟
- \(2o\) 是发送端和接收端各一次 CPU 处理开销
- \((m-1) \times G\) 是数据传输时间

对于小消息，\(L\) 和 \(o\) 主导；对于大消息，\(G \times m\) 主导。

### 7.4 协议切换

现代网络协议通常有两种模式：

- **Eager 协议**（小消息）：发送端直接发送，无需事先握手。延迟低但占用接收端缓冲区。
- **Rendezvous 协议**（大消息）：发送端先通知接收端"准备好了吗？"，收到确认后才发送。增加了一个往返延迟，但内存使用更可控。

SysSim 的 LogGP 参数加载器支持根据消息大小自动选择对应的协议参数。

### 7.5 参数获取

LogGP 参数通过**乒乓微基准测试**（Ping-Pong Microbenchmark）获得：在两个节点之间反复发送不同大小的消息，测量往返时间（RTT），然后通过最小二乘拟合提取 \(L\)、\(o\)、\(g\)、\(G\) 参数。

## 8. 设备网格与分布式训练

### 8.1 为什么需要分布式训练？

现代 LLM 的规模已经远超单块 GPU 的内存容量。例如，一个 70B 参数的模型仅存储参数就需要约 140 GB（FP16），而当前最大的单 GPU 内存也只有 80-96 GB。因此，**分布式训练**（Distributed Training）——使用多块 GPU 协同训练——是必不可少的。

### 8.2 三种核心并行策略

**数据并行（Data Parallelism, DP）**

每块 GPU 持有完整的模型副本，但处理不同的数据批次。训练后通过 AllReduce 同步梯度。

```
           数据批次
      ┌────┬────┬────┬────┐
      │ D0 │ D1 │ D2 │ D3 │
      └──┬─┴──┬─┴──┬─┴──┬─┘
         ▼    ▼    ▼    ▼
      ┌────┐┌────┐┌────┐┌────┐
      │GPU0││GPU1││GPU2││GPU3│  每块 GPU 有完整模型
      │模型││模型││模型││模型│  处理不同数据
      └──┬─┘└──┬─┘└──┬─┘└──┬─┘
         │    │    │    │
         └────┴────┴────┘
              AllReduce           同步梯度
```

优点：实现简单，通信量与模型大小成正比（与 GPU 数量无关）。
缺点：每块 GPU 必须能容纳完整模型。

**张量并行（Tensor Parallelism, TP）**

将单个算子（如矩阵乘法）的计算**切分**到多块 GPU 上。每块 GPU 只持有权重矩阵的一部分。

```
  完整权重矩阵 W (大小 M×N):

  ┌──────────────────────────────┐
  │            W                 │
  │         M × N                │
  └──────────────────────────────┘
              │
    按列切分为 4 份
              │
  ┌──────┐┌──────┐┌──────┐┌──────┐
  │ W_0  ││ W_1  ││ W_2  ││ W_3  │
  │M×N/4 ││M×N/4 ││M×N/4 ││M×N/4 │
  │GPU 0 ││GPU 1 ││GPU 2 ││GPU 3 │
  └──────┘└──────┘└──────┘└──────┘
```

优点：可以训练超大模型（单 GPU 内存不足时）。
缺点：每层计算后需要 AllReduce 或 AllGather 通信，要求 GPU 之间有**高带宽互联**（如 NVLink）。

**流水线并行（Pipeline Parallelism, PP）**

将模型的不同**层**分配到不同的 GPU 上，数据像流水线一样依次流过各 GPU。

```
  GPU 0      GPU 1      GPU 2      GPU 3
  ┌─────┐   ┌─────┐   ┌─────┐   ┌─────┐
  │第1-6│──►│第7-12│──►│13-18│──►│19-24│
  │  层  │   │  层  │   │  层  │   │  层  │
  └─────┘   └─────┘   └─────┘   └─────┘
```

优点：通信量小（只在层间传递激活值）。
缺点：存在"流水线气泡"——部分 GPU 在等待数据时空闲。

### 8.3 设备网格（Device Mesh）

**设备网格**是一种将多块 GPU 组织成多维逻辑阵列的抽象。每个维度对应一种并行策略。

例如，16 块 GPU 排列为 \(4 \times 4\) 的二维网格：

```
             GPU 维度（张量并行）
            0    1    2    3
          ┌────┬────┬────┬────┐
  节点 0  │ G0 │ G1 │ G2 │ G3 │
  维度    ├────┼────┼────┼────┤
 （数据  1│ G4 │ G5 │ G6 │ G7 │
  并行）  ├────┼────┼────┼────┤
        2 │ G8 │ G9 │G10 │G11 │
          ├────┼────┼────┼────┤
        3 │G12 │G13 │G14 │G15 │
          └────┴────┴────┴────┘

  同一行内（如 G0, G1, G2, G3）：张量并行组，通过 NVLink 通信
  同一列内（如 G0, G4, G8, G12）：数据并行组，通过 InfiniBand 通信
```

### 8.4 网络拓扑

不同维度的 GPU 之间通常使用不同的互联技术：

**NVLink Mesh**（节点内，同一台机器的 GPU 之间）：
- 带宽极高（例如 900 GB/s）
- 延迟极低
- 适合张量并行等高频通信

**InfiniBand / Slingshot**（节点间，不同机器之间）：
- 带宽相对较低（例如 25-100 GB/s）
- 延迟较高
- 适合数据并行等通信量较大但频率较低的场景

SysSim 支持五种拓扑模型：

| 拓扑 | 描述 | 典型应用 |
|------|------|----------|
| FullyConnected | 每对节点有专用链路 | 理论基准 |
| Ring | 双向环形 | 环形 AllReduce |
| Switch | 星形，共享交换机 | 简单集群 |
| NVLinkMesh | 全连接 NVLink 网格 | 节点内 GPU |
| Hierarchical | 多层（NVLink + InfiniBand） | 生产集群 |

### 8.5 最大-最小公平带宽共享

当多个通信操作同时使用同一网络资源时，SysSim 使用**最大-最小公平**（Max-Min Fairness）算法分配带宽：

$$\text{bandwidth}_i = \min_{r \in \text{path}_i} \frac{\text{bandwidth}_r}{\text{num\_ops\_sharing}_r}$$

即每个操作获得其路径上最拥挤资源的公平份额。这自然地模拟了网络拥塞。

## 9. CUDA Stream 与同步机制

### 9.1 什么是 CUDA Stream？

**CUDA Stream** 是 GPU 上的一个**任务队列**。提交到同一个 Stream 的操作按顺序执行，但不同 Stream 上的操作可以**并行**执行。

可以类比超市收银：
- 每条收银通道是一个 Stream
- 同一通道的顾客按顺序结账
- 不同通道的顾客同时结账

```
Stream 0:  ████ Op A ████ ──── ████ Op C ████
Stream 1:       ████ Op B ████ ──── ████ Op D ████

时间 ──────────────────────────────────────────►

Op A 和 Op B 可以并行执行（不同 Stream）
Op C 必须等 Op A 完成（同一 Stream）
Op D 必须等 Op B 完成（同一 Stream）
```

### 9.2 为什么要使用多个 Stream？

在 LLM 训练中，许多操作之间没有数据依赖关系，可以并行执行：

- **计算与通信重叠**：当 GPU 在进行第 N 层的反向传播计算时，可以同时在另一个 Stream 上传输第 N+1 层的梯度。
- **多算子并行**：独立的数学运算可以分配到不同 Stream 并行执行。

### 9.3 跨 Stream 同步

当一个 Stream 的操作依赖于另一个 Stream 的结果时，需要**同步**。CUDA 提供了 **Event** 机制：

```
Stream 0:  ████ Op A ████ ─── record(Event) ───── ████ Op C ████
                                    │
                                    ▼
Stream 1:  ──── wait(Event) ─── ████ Op B ████

Op B 需要等待 Op A 完成（通过 Event 同步）
Op C 不受影响，继续在 Stream 0 上按序执行
```

SysSim 通过以下两种机制追踪这些同步关系：

- **BARRIER（栅障）**：等待**所有** Stream 上的操作完成（全局同步点）
- **STREAM_SYNC（流同步）**：只等待**特定** Stream 上的操作完成（定向同步）

### 9.4 SysSim 中的追踪实现

SysSim 通过拦截 PyTorch 的分发机制来追踪 Stream 操作：

- **TensorStorageTracker**：跟踪张量的存储指针，建立"谁产生了这个张量"的映射，从而推断数据依赖。
- **CUDAEventTracker**：通过猴子补丁（Monkey Patching）拦截 `Event.record()` 和 `Event.wait()` 调用，捕获跨 Stream 的同步关系。

### 10.3 适应新硬件

SysSim 的一个重要设计目标是**硬件可移植性**：

| 组件 | 新硬件需要改什么？ | 工作量 |
|------|---------------------|--------|
| Roofline 模型 | 只改硬件参数（Peak FLOP/s、带宽等） | 几分钟 |
| ML 效率预测器 | 需要在目标硬件上重新分析（Profiling）并训练 | 几分钟到几小时 |
| FLOP 计数 | 无需改动 | 零 |
| 网络模拟器 | 更新 LogGP 参数（通过微基准测试获取） | 几分钟 |
| 追踪器 | 无需改动 | 零 |
